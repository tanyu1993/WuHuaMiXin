# 2026-03-05 开发日志：资源估价系统与全图鉴补全功能

## 1. 核心需求变更
*   **全图鉴模式**：解决用户截图不全导致录入漏掉器者的问题。开启后，系统自动补全所有未识别的“特出/限定”器者，状态设为拥有，致知设为0。
*   **资源价值评估**：将账号内未使用的抽卡资源（灰珀、彩珀、券）和致知资源（通行红卡、红票）以及月卡天数折算为人民币价值，计入总估值。
*   **配置化管理**：资源估价的系数（如单抽价格、红卡折合抽数、活资源溢价等）需在“估值模型配置”中可手动调节。

## 2. 估值模型逻辑达成一致
基于市场调研（单抽约 0.15 元）及“活资源”灵活性考虑，确立以下公式：

*   **活资源溢价系数 (`LIVE_RESOURCE_BONUS`)**：默认 **1.2**。由于货币和红卡可以留给未来更强的限定，且不会像已有角色那样随版本贬值，其价值高于已消耗资源。
*   **抽卡资源**：`等效抽数 = 灰珀/150 + 彩珀/3 + 请调券`。
*   **致知资源（红卡）**：按照 T0 (1.6) 和 T1 (0.7) 的平均强度权重计算。
    *   `1张红卡 = 0.25个本体增量 ≈ 8.6抽`（基于 60 抽期望及致知价值增量推导）。
*   **月卡资源**：`每日价值 ≈ 0.8元`（按 30 元/月，8 折预付费折扣计）。

## 3. 技术实施记录

### 3.1 前端 UI 修改
*   **`config.html`**：新增“额外资源价值参数”模块，支持调节 `LIVE_RESOURCE_BONUS`、`RED_CARD_TO_PULLS` 和 `DAILY_MONTHLY_CARD_VAL`。
*   **`batch_audit.py` (HTML 生成)**：
    *   在校验页面第一步新增 `resource-panel` 面板，包含 6 个数值输入框。
    *   新增“✨ 开启全图鉴模式”勾选框。
    *   美化了资源录入区的 CSS 布局。

### 3.2 后端逻辑修改
*   **`app.py`**：
    *   更新 `finish()` JS 函数，使之能封装 `characters` 和 `resources` 嵌套 JSON 结构。
    *   调整 `/save` 路由：若开启全图鉴，调用 `db.get_up_pool_names()` 获取所有 UP 器者并为缺失项补齐致知 0。
    *   优化 `/config` 路由，确保嵌套字典（强度/致知权重）与扁平参数能同步更新。
*   **`valuation_engine.py`**：
    *   重构 `calculate_account_value`：将器者估值（受贬值和图鉴系数影响）与资源估值（独立计算，不打折）分离后汇总。
    *   在报告 `top_assets` 明细中增加资源折算项及其计算公式展示。

### 3.3 稳定性与 Bug 修复
*   **CSS 语法冲突**：修复了 Python f-string 渲染 HTML 时因单大括号 `{}` 导致的 `NameError`，已全部替换为 `{{}}` 转义。
*   **PaddleOCR 并发异常**：针对 `RuntimeError: could not execute a primitive` 报错，引入了 `threading.Lock` 全局锁，并设置 Flask 为 `threaded=False` 单线程模式，确保 OCR 引擎运行稳定。

## 4. 后续建议
*   目前的红卡折合抽数（8.6 抽）是基于 T0/T1 平均权重。如果未来 T0 限定极其强势，可以考虑在配置页手动调高此系数。
*   全图鉴模式会补全所有“值得录入”的红卡，如果用户某些常驻红卡也未抽到，该功能也会将其补为致知 0。
