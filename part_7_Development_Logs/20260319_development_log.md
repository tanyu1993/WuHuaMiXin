# 📅 2026-03-19 开发日志：从正则“缩圈”到“机助人校”的战略突围

## 1. 战略转型：从“技能打标”向“状态建模”的撤退与反攻
*   **挑战复盘**：初期尝试通过正则表达式（Regex）对 8000+ 条非标技能描述进行“缩圈”打标。虽历经 V2.2 至 V5.1 五次逻辑迭代，解决了跨句范围继承、状态隔离等难题，但由于文案随意性极高，正则边际收益迅速递减。
*   **重大决策**：果断放弃对“表现层”（技能文本）的直接解析，转而深耕“协议层”（状态系统）。
*   **逻辑支撑**：状态（Status）是游戏数值设计的原子单位，全量约 300+，且描述极其规范。通过打透状态库，可实现“基于战术收益反向寻找器者”的图鉴核心功能。

## 2. 战术建模：物华弥新“原子化”标签体系 (V2.0 - V2.6)
*   **多维演进**：设计了涵盖“输出强化、易伤加深、战术循环、硬性控制、生存韧性”的 5 大核心战术域。
*   **语义纠偏关键点**：
    *   **攻防隔离**：引入邻近匹配（Proximity Match），彻底隔离了“受到的伤害降低（增益）”与“造成伤害降低（减益）”。
    *   **噪音抑制**：通过“词组绑定”解决了“持续 X 回合”与“持续伤害（DoT）”的语义冲突。
    *   **名即是标**：确立了“状态名匹配优先”原则，大幅提升了核心机制（回溯、援护、额外伤害）的识别精度。

## 3. 工程化基建：校准中心 V3.1 (Human-in-the-loop)
*   **Flask 后端增强**：新增 `/tagger` 路由及 `/api/verify_status` 接口，支持标签增删、状态固化（Verified）的双向同步。
*   **可视化 Dashboard**：
    *   实现了数据驱动的交互界面，支持实时筛选、手动更正/删除标签。
    *   **引擎熔断机制**：引入 `verified` 标志位。一旦人工固化，正则引擎严禁修改该节点，确保人工资产不被自动化逻辑冲刷。

## 4. 危机处理：浏览器内存紧急救援手术
*   **事故描述**：由于打标引擎 V2.5 存在“非固化即覆盖”的逻辑漏洞，导致约 20+ 个手动录入的新标签在同步时被正则逻辑误洗。
*   **救援方案**：利用浏览器未刷新的特性，通过 Console 控制台导出 `STATUS_DATA` 内存镜像，生成 `recovered_status_data.json`。
*   **成果恢复**：编写 `final_recovery.py` 进行数据“缝合”，成功找回 274 条状态记录，标签种类数从 56 种恢复并增长至 **83 种**。

## 5. 今日里程碑 (Milestones)
*   **数据确权**：已人工固化 **142 个** 核心状态，系统进入“机助人校”的高速成长期。
*   **标签沉淀**：确立了 **83 种** 符合《物华弥新》实战逻辑的精准战术标签。
*   **架构稳固**：打标引擎升级至 **V3.2 (保护版)**，确立了“手动修改优先、固化状态熔断、正则仅补空位”的最终运行准则。
