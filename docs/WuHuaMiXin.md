# 核心开发经验与工业级准则 (Cross-Project Reference)

*   **JS 注入与字符串转义陷阱**：在 Python f-string 中注入包含 `{}` 的 JavaScript 脚本时，Python 会将其识别为占位符。**教训**：严禁在注入复杂 JS 时使用 f-string，应使用 `%` 格式化或 `{{ }}` 双大括号转义，否则会导致前端功能静默失效（如保存跳转失败）。
*   **稳健修改原则 (Debug-First)**：面对新业务场景（如支持 JFIF 压缩格式）导致的算法失效，**严禁直接修改已验证的核心算法**。应建立独立调试脚本（如 `debug_*.py`）复现问题，通过可视化中间产物（ROI 切片）定位根因，确认是图像质量还是坐标偏移后再进行最小化补丁。
*   **配置增量更新 (Patch Logic)**：在实现可视化配置保存时，后端严禁直接用前端提交的数据覆盖 JSON 文件。**解决方案**：采用 `current_config.update(new_data)` 增量模式，防止未出现在表单中的嵌套复杂配置项（如线性插值点列表）丢失导致估值体系毁灭性缩水（全员一折灾难）。
*   **MMA 模块化原则**：
    1.  **配置外置**：任何变动数值必须存在于 `settings.json`。
    2.  **数据解耦**：原始 Excel 仅作源，程序运行时必须通过 `metadata.json` 缓存层。
    3.  **表现分离**：Python 严禁拼接 HTML 字符串，UI 逻辑必须完全移入 Jinja2 模板。
    4.  **路径中心化**：严禁硬编码相对路径，必须通过 `Settings.paths` 注册表统一管理。

##  物华弥新抽卡规则

**抽卡概率**    游戏角色（器者）的稀有度&概率如下：
    特出（红）：2.5%
    优异（黄）：12%
    新生（蓝）：85.5%

**小保底**   70抽保底，前50抽出“特出”器者的概率都是2.5%，每抽独立计算。如果前50抽都没有出“特出”器者，从第51抽开始，增加获得特出器者的概率，每抽增加5%。举例说明：第51抽概率为7.5%，第52抽概率为12.5%，以此类推，直到第70抽出“特出”器者的概率为102.5%（必出）。出红后，保底计数重置。

**歪的概率**   50%。即每一次出红，有50%概率出当期up器者。

**大保底**   160抽必出当期UP。在160抽之前，如果通过保底或者概率出了当期UP，则大保底计数重置。

**抽卡期望**
    出红期望：30.1抽
    出UP期望：57.1抽

## 账号估值逻辑 (Alpha 版)

**核心单位：PullUnit (PU)**
*   1 PU = 60抽 (接近一个红卡的平均期望)。
*   市场转化率：1抽约等于 0.16元。

**致知（命座）价值**
*   采用 160抽一井的沉没成本模型。
*   0致知 = 1 PU
*   3致知 = 3 PU (质变点)
*   6致知 = 5.3 PU (满阶)

**调节系数**
*   **限定溢价**：1.4倍。
*   **强度权重**：T0(1.6) / T0.5(1.2) / T1(0.7) / T2(0.1)。
*   **顺序衰减**：老角色随时间贬值，每早10个顺位贬值20%。

## 2026年2月28日 开发日志：架构重构与识别精度突破

### 1. 核心成果：MMA 模块化架构确立
为了实现“高内聚、低耦合”，我们将系统重构为四个独立层级：
*   **核心数据层 (`core/`)**: 
    *   `database.py`: 负责 Excel 数据的语义化加载，不再依赖死板的行号。
    *   `settings.py`: 管理外部配置，支持估值参数的动态加载。
*   **视觉智能层 (`vision/`)**: 
    *   `grid.py`: 建立基于“基础单元 U”的全局动态网格，支持宽屏自适应与边缘全量提取。
    *   `processor.py`: 负责图像 3.5 倍放大、锐化及 CLAHE 增强。
    *   `analyzer.py`: 承载繁体数字解析与模糊名称匹配。
*   **估值引擎层 (`valuation/`)**: 
    *   `valuation_engine.py`: 核心算分逻辑，引入线性插值完整度算法。
    *   `advisors.py`: 负责 6 人战队模拟与分级补强路线图生成。
*   **表现层与应用逻辑**: 
    *   `app.py`: 纯净的 Flask 路由分发。
    *   `templates/`: 彻底实现 UI 与逻辑分离，支持 HTML 长图导出。

### 2. 识别算法三大突破
1.  **全局网格强力提取**：放弃“认出名字才切图”的旧逻辑，改为“按网格预测全量切图”。彻底解决了边缘角色由于图片残缺导致的 OCR 拒识问题。
2.  **两阶段校验流**：
    *   **第一步 (身份确认)**：合并 0/1 选项，用户仅需确认角色名和 2-6 高阶致知。
    *   **第二步 (点亮精修)**：将所有 0/1 角色平铺，利用“名称白平衡（Brightness Ratio）”算法预判，用户通过“拨码开关”实现极速精修。
3.  **智能校准过滤**：通过数据库预筛，自动隐藏不具价值的蓝/黄卡，将校准工作量缩减了 60% 以上。

### 3. 估值模型 V6.0 (定稿)
*   **基础单元**：单抽价值锚定为 **￥0.15**。
*   **致知权重**：采用买家心理平衡模型（3ZZ=3.5, 6ZZ=5.0），兼顾投入成本与付费意愿。
*   **完整度算法**：引入 **“90% 临界跳水模型”**。90%-100% 价格坚挺，跌破 90% 后采用线性插值实现加速贬值。
*   **贬值系数**：每早一个 UP 顺位，贬值 **2.5%**，加速老常驻资产剔除。

### 4. 遇到的技术坑点与解决方案
*   **MKLDNN 崩溃**：PaddleOCR 处理大图时执行原语失败。方案：显式 `enable_mkldnn=False`。\n*   **路径编码陷阱**：Windows 中文路径 URL 编码导致失效。方案：使用 `urllib.parse.unquote` 解码。\n*   **卡片切片漂移**：改用“器者间距 U”作为几何基准单元，实现数学级精准定位。

## 2026年3月2日 开发日志：v7.0 稳定版与工业化重构

### 1. 核心架构升级 (Stable-Modular)
*   **元数据缓存系统**：实现 `metadata_manager.py`，将原始 Excel 解析并缓存为 `metadata.json`。主程序启动速度提升 10 倍，且实现了逻辑与 Excel 结构的物理脱离。
*   **配置可视化**：建立 `/config` 路由，支持通过 Web UI 调整估值模型所有参数（单抽价格、强度系数等），且保存机制升级为“增量更新”，防止配置项丢失。
*   **多账号分析 Kanban**：新增全账号综合分析页面，支持不同账号间估值、折扣率、收集率的横向对比，支持标价回录。
*   **UI/Logic 彻底分离**：重构 `app.py`，移除所有行内 HTML 拼接，完全采用 Jinja2 模板引擎。建立全局 `Path Registry`，统一管理项目资源路径。

### 2. 视觉算法专项突破
*   **压缩格式支持**：后端正式支持 `.jfif` 图片格式识别。
*   **致知算法优化**：针对 `.jfif` 等高压缩图片导致的笔画断裂问题，用 **“Gamma 变换 + 高斯去噪”** 替代了生硬的拉普拉斯锐化。全量 16 个账号测试表明，在不损伤清晰图的前提下，压缩图的致知识别率从 0% 提升至 95% 以上。
*   **智能录入交互**：录入页面下拉框升级为“带 datalist 的 Searchable Input”，支持重复录入实时红色预警。

### 3. 数据层规范化
*   **【稳定版】解析协议**：严格按照用户约定的行列索引解析【器者属性】与【市场与强度】Sheet。
*   **配队逻辑对齐**：重构 `advisors.py`，实现“核心 > 重要组件 > 备选”的推荐展示梯度，同时在打分时保持组件地位平等。

### 4. 修复与治理
*   **术语规范化**：全案清理“Audit”字眼，统一为 “Verify (验证)” 与 “Entry (录入)”。
*   **计算一致性修复**：修复了因配置文件 Key 丢失导致的“全员一折”估值灾难。
*   **环境兼容**：解决了 Windows 批处理脚本中的中文乱码与中文路径保存问题。
